Containers in a DevOps Workflow

Connect Docker to your CI pipeline and complete the journey from git push to a verified, containerised application.

MLO5MLO6MLO7

Containers in the pipeline

In Module 8 you built a Docker image locally. Now let us connect that to the CI pipeline you built in Modules 6 and 7.

The idea is simple: after the tests pass, the pipeline should also verify that the Docker image can be built and that the container runs correctly. This means if your Dockerfile has an error, you find out immediately — not when someone tries to deploy the application.

Why build in CI?

Building the image locally is useful. Building it in CI is essential. CI gives you a clean, neutral environment. If it builds and runs there, it will build and run anywhere.

Adding docker build to CI

GitHub's hosted runners have Docker pre-installed. Add two steps to your workflow:

.github/workflows/ci.yml — docker steps
      - name: 
Run tests
        run: 
./run.sh test

      - name: 
Build Docker image
        run: 
docker build -t my-app:latest .

      - name: 
Run container smoke test
        run: 
docker run --rm my-app:latest

The --rm flag tells Docker to automatically remove the container after it exits. This keeps the runner clean.

A smoke test is a minimal check — it does not test every feature, it just confirms the application starts and exits without crashing. This catches the most common deployment errors.

Environment variables in containers

Applications often need configuration that varies between environments — a database URL, an API key, a flag to enable debug mode. You should never hard-code these values in the image. Instead, pass them at runtime:

bash — passing environment variables
# Pass a single variable with -e
docker run 
-e 
APP_ENV=production 
my-app

# Pass multiple variables
docker run 
-e 
APP_ENV=test 
-e 
DEBUG=false 
my-app

In CI, you can pass the same variable from the workflow:

.github/workflows/ci.yml — env in docker run
      - name: 
Run container smoke test
        run: 
docker run --rm -e APP_ENV=test my-app:latest

The full pipeline story

Let us trace the complete journey from a developer's commit to a verified, containerised application:

  1. Developer pushes code — git push origin feature-branch
  2. Pipeline triggers — GitHub Actions detects the push and starts the workflow.
  3. Checkout — the runner checks out the code.
  4. Install and test — ./run.sh install then ./run.sh test run the automated checks.
  5. Build image — docker build -t my-app:latest . packages the application.
  6. Smoke test — docker run --rm my-app:latest confirms the container starts cleanly.
  7. Pipeline passes — the PR can now be reviewed and merged with confidence.

This is a real, working DevOps pipeline. It is not the most complex one in the world, but it embodies every principle from the CALMS framework: automation, consistency, fast feedback.

Common deployment challenges

Containers make deployment more predictable, but some challenges remain:

Port conflicts
Two processes trying to use the same port. Solved by mapping container ports explicitly.
Missing env vars
The app needs a variable that was not passed at runtime. Fails loudly at startup.
Image not rebuilding
Docker caches layers aggressively. Sometimes you need --no-cache to force a fresh build.
Large images
Installing too many packages inflates the image. Use slim base images and clean up after RUN steps.
Stopped vs running
A container that exits immediately is not the same as one that never started. Check docker ps -a.
The end of this module is not the end of the story

This module deliberately stopped before container orchestration (Kubernetes), container networking, volumes, and registry deployments. These are real and important topics — they are covered in later modules and in industry courses. The foundation you have built here is the necessary prerequisite for all of them.

Key terms

smoke test
A minimal check that the application starts and runs without immediately crashing.
--rm
A docker run flag that removes the container automatically when it exits.
-e
A docker run flag that passes an environment variable to the container.
image tag
A label for a specific version of an image, e.g. my-app:latest.
docker build cache
Docker reuses unchanged layers to speed up builds. Use --no-cache to bypass.
pipeline parity
Local docker build and CI docker build produce identical images.

Exercises

Lab exercises
  1. Write a Dockerfile for the application from Module 6 and 7.
  2. Build and run it locally — confirm the output matches running the script directly.
  3. Add a docker build step to your GitHub Actions workflow.
  4. Add a docker run --rm step as a smoke test.
  5. Push and verify the complete pipeline passes: install, test, build, run.
  6. Extension: pass an environment variable APP_ENV=test to the container in CI, and print it in your Python script to confirm it is received.
