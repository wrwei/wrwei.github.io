Automating Repetitive Tasks

Stop typing the same commands every day. Write automation scripts that work identically locally and in CI.

MLO3MLO6

Why automate?

Consider what you do every time you start working on a project: navigate to the folder, activate a virtual environment, maybe install updated packages, run a quick test to make sure everything is still working.

These are the same steps every single time. If you do something more than twice, it should be automated. There are two reasons:

  • Consistency — a script runs the same steps in the same order every time. You cannot forget a step or do it slightly differently.
  • Shareability — a script communicates the correct workflow to everyone on the team, including your future self.
The connection to CI

Your CI pipeline is just automation running on someone else's computer. When you automate your local workflow with a script and then call that same script from the pipeline, you are guaranteed that local and CI behave identically. This is exactly the goal.

Identifying repetitive tasks

Look at your workflow from the previous modules. You have been typing something like this every session:

bash — manual workflow (tedious)
cd 
~/devops-labs/my-project
pip install 
-r 
requirements.txt
flake8 
.
pytest

Four commands, every time. If you add a new tool later, you have to remember to run it. If someone new joins the project, they have to figure out the workflow from scratch.

A script solves all of this:

bash — run.sh
#!/bin/bash
set 
-e
   # stop immediately if any command fails

echo 
"Installing dependencies..."
pip install 
-r 
requirements.txt

echo 
"Linting..."
flake8 
.

echo 
"Running tests..."
pytest

echo 
"All checks passed."

A well-structured automation script

As scripts grow, give them structure. A common pattern is to define named tasks as functions, then call the right one based on an argument:

bash — structured tasks script
#!/bin/bash
set 
-e

install() {
    echo 
"Installing..."
    pip install 
-r 
requirements.txt
}

test() {
    echo 
"Testing..."
    pytest
}

clean() {
    echo 
"Cleaning..."
    rm 
-rf 
__pycache__ .pytest_cache
}

# Call the function matching the argument
"$@"

Now you can call ./run.sh install, ./run.sh test, or ./run.sh clean individually.

Connecting scripts to CI

Replace inline commands in your pipeline with calls to your script:

.github/workflows/ci.yml — calling the script
      - name: 
Install dependencies
        run: 
./run.sh install

      - name: 
Run tests
        run: 
./run.sh test

This has a valuable property: the pipeline now does exactly what a developer would do locally. If you change the workflow, you update the script — and both local and CI benefit automatically.

Make the script executable

GitHub Actions runners start fresh each time. The script must be committed with execute permissions: chmod +x run.sh, then git add run.sh and commit.

Worked example

Here is the full picture: a developer works on a feature, runs the local script to verify everything is clean, pushes, and the pipeline calls the same script:

bash — developer's local session
# Make changes to the code
vim 
app.py
# Run the automated checks
./run.sh test
Testing...
3 passed in 0.11s
# Everything passes — commit and push
git add 
.
git commit 
-m 

git push
# GitHub Actions now runs the same ./run.sh test

Key terms

automation script
A shell script that performs a named task: install, test, clean.
set -e
A shell option that stops the script immediately if any command fails.
function
A named block of commands in a shell script, called by name.
$@
A special variable in bash meaning 'all the arguments passed to this script'.
idempotent
Running the same script twice produces the same result as running it once.
workflow parity
Local and CI use the same scripts so they behave identically.

Exercises

Lab exercises
  1. Write a script run.sh with three tasks: install, test, and clean.
  2. Add set -e at the top. Deliberately make one step fail — does the script stop?
  3. Run each task locally: ./run.sh install, ./run.sh test.
  4. Update your GitHub Actions workflow to call the script instead of running commands directly.
  5. Push and confirm the pipeline still passes.
  6. Extension: add a task that checks whether Python 3 is installed before doing anything else, printing a helpful error message if not.
