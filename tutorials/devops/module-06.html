CI Pipelines in Practice with GitHub Actions

Understand pipeline structure and learn to read, modify and debug a real GitHub Actions workflow.

MLO2MLO4MLO5

Anatomy of a pipeline

A CI pipeline is a sequence of steps that run automatically when code is pushed. Every pipeline has the same basic anatomy:

Trigger
The event that starts the pipeline — usually a push or a pull request.
Job
A unit of work that runs on a server. A pipeline can have one or more jobs.
Step
A single command or action within a job.
Runner
The server that executes the job. GitHub provides free hosted runners.
Artefact
A file produced by the pipeline — for example, a test report.

For this module, every pipeline will be a single job with a linear sequence of steps: install dependencies, lint, run tests.

GitHub Actions concepts

GitHub Actions is the CI system built into GitHub. When you push to a repository, it looks for workflow files in the .github/workflows/ folder.

A workflow file is written in YAML — a human-readable format that uses indentation to express structure. Here is a minimal but complete example:

.github/workflows/ci.yml
name: 
CI

on:
  push:
    branches: 
[main]
  pull_request:

jobs:
  test:
    runs-on: 
ubuntu-latest

    steps:
      - name: 
Check out code
        uses: 
actions/checkout@v4

      - name: 
Set up Python
        uses: 
actions/setup-python@v5
        with:
          python-version: 
'3.11'

      - name: 
Install dependencies
        run: 
pip install -r requirements.txt

      - name: 
Run tests
        run: 
pytest

Reading a workflow file

Let us trace through the workflow above line by line:

  1. name: CI — the display name shown in the GitHub Actions tab.
  2. on: push / pull_request — this pipeline runs whenever someone pushes to main or opens a pull request.
  3. jobs: test: — defines one job called test.
  4. runs-on: ubuntu-latest — the job runs on a fresh Ubuntu Linux server provided by GitHub.
  5. steps: — the sequence of things the job does.
  6. uses: actions/checkout@v4 — a pre-built action that checks out your code so the runner can access it.
  7. run: pip install -r requirements.txt — install Python packages listed in your requirements file.
  8. run: pytest — run all tests.
YAML indentation matters

In YAML, indentation defines structure. Two spaces is the convention. Never mix tabs and spaces — it will cause a parse error and the pipeline will not run.

Modifying the workflow

Rather than writing a workflow from scratch, you will work from the provided template. The most common modifications you will make:

Change the Python version:

.github/workflows/ci.yml — change Python version
        python-version: 
'3.12'
        # change '3.11' to any supported version

Add a linting step before the tests:

.github/workflows/ci.yml — add a lint step
      - name: 
Lint with flake8
        run: 
pip install flake8 && flake8 .

Add an environment variable available to all steps:

.github/workflows/ci.yml — environment variable
    env:
      APP_ENV: 
test
      DEBUG: 
false

Observing the pipeline

After you push, navigate to your repository on GitHub and click the Actions tab. You will see:

  • A list of recent workflow runs — green for pass, red for fail
  • Click any run to see the jobs and steps
  • Click a step to expand its log output

When a run fails, the failed step is highlighted. Click it to see the exact error. This is the most important skill in working with CI: reading the log efficiently.

Your first pipeline will probably fail

Almost everyone's first pipeline fails — usually due to a missing dependency in requirements.txt, a wrong path, or a YAML indentation error. This is normal. Read the error, fix it, push again.

Key terms

GitHub Actions
The CI/CD system built into GitHub, configured with YAML workflow files.
.github/workflows/
The folder where GitHub Actions looks for workflow files.
YAML
A human-readable data format using indentation for structure.
trigger (on:)
The event that causes the workflow to run.
uses:
Runs a pre-built action from the GitHub Marketplace.
run:
Executes a shell command directly on the runner.

Exercises

Lab exercises
  1. Fork the provided starter repository — it contains a Python script and a test file.
  2. Look at the workflow file in .github/workflows/. Trace what each step does.
  3. Add a step to install dependencies from requirements.txt before the tests run.
  4. Push and check the Actions tab — did the pipeline pass?
  5. Open test_app.py and deliberately break one test (change an expected value). Push and observe the failure.
  6. Read the error in the Actions log. Fix the test, push, and confirm it passes.
  7. Extension: add a linting step using flake8. Introduce a style error and observe it being caught.
