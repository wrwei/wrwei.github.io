Virtualisation and Containers

Understand the 'works on my machine' problem and learn to package applications with Docker.

The environment problem

Here is a scenario that happens to almost every developer:

The classic problem

You spend an afternoon writing a script that works perfectly. The next day you send it to a classmate. They run it and get an error you have never seen. After an hour of debugging, you discover they have a different version of Python installed. Or a missing library. Or a different operating system.

"It works on my machine" is one of the most common phrases in software development — and one of the most frustrating.

The fundamental issue is that software depends on its environment — the operating system, the language version, the installed packages. Environments differ between machines, and those differences cause unpredictable failures.

Virtual machines vs containers

Two technologies address this problem in different ways:

A virtual machine (VM) solves the problem by running a complete, separate operating system inside your computer. Your Python code runs on this 'guest' OS, which is identical for everyone. VMs work well, but they are large (often several gigabytes) and slow to start.

A container is a lighter approach. Instead of running a whole separate OS, it isolates just the application and its dependencies — the Python version, the libraries — and shares the host OS's kernel. Containers start in seconds and use much less disk space.

VM
A full virtual computer running its own OS. Heavyweight but very isolated.
Container
An isolated application environment sharing the host OS kernel. Lightweight and fast.
Docker
The most widely used tool for building and running containers.
image
A blueprint for a container — a read-only snapshot of an application and its environment.
container
A running instance of an image. You can run many containers from one image.

Docker fundamentals

Docker has three main concepts:

  1. Image — a read-only template that describes the environment. Think of it as a recipe.
  2. Container — a running instance of an image. A recipe can produce many cakes; an image can produce many containers.
  3. Registry — a place to store and share images. Docker Hub is the public registry; most official images (Python, nginx, postgres) are available there.

The core workflow is: write a Dockerfile → build an image → run a container.

Your first Dockerfile

A Dockerfile is a text file that describes how to build an image. Each line adds a layer:

Dockerfile
# Start from an official Python image
FROM 
python:3.11-slim

# Set the working directory inside the container
WORKDIR 
/app

# Copy your requirements file and install dependencies
COPY 
requirements.txt .
RUN 
pip install -r requirements.txt

# Copy the rest of your code
COPY 
. .

# The command to run when the container starts
CMD 
["python", "app.py"]

Each instruction builds on the previous one. The FROM line is always first — it is the base image you start from. Choosing a -slim variant keeps the image small.

Core Docker commands

bash — essential Docker commands
# Build an image from the Dockerfile in the current directory
docker build 
-t 
my-app:latest .

# List all local images
docker images
REPOSITORY   TAG       IMAGE ID       SIZE
my-app       latest    a1b2c3d4e5f6   142MB

# Run a container from your image
docker run 
my-app:latest

# Run and get a shell inside the container
docker run 
-it 
my-app:latest bash

# List running containers
docker ps

# Stop a running container
docker stop 


# Remove a stopped container
docker rm
The -t flag

The -t flag gives your image a name and optional tag: name:tag. Tags are version labels. latest is used by convention for the most recent version.

Key terms

Dockerfile
A text file of instructions that builds a Docker image.
FROM
The base image your Dockerfile builds on.
WORKDIR
Sets the working directory inside the container.
COPY
Copies files from your local machine into the image.
RUN
Executes a command during the image build (e.g. install packages).
CMD
The default command to run when a container starts.

Exercises

Lab exercises
  1. Pull and run the official Python image: docker run -it python:3.11-slim bash. You are now inside a container — explore it.
  2. Exit with exit. Run docker ps -a — you should see the stopped container.
  3. Write a Dockerfile for a Python script that prints 'Hello from Docker'.
  4. Build it: docker build -t hello-docker .
  5. Run it: docker run hello-docker
  6. Edit the Python script (outside the container), rebuild the image and run again.
  7. Extension: use docker history my-app to see the layers in your image.
