The pipeline end to end

Before revising individual topics, let us trace one complete scenario from start to finish — this is the kind of worked example that appears in exams.

Scenario

A developer notices a bug: the login function returns True even when the password is wrong. They fix it, test it locally, and push to GitHub. Walk through what happens.

  1. Fix and commit — the developer edits auth.py and commits: git commit -m "Fix login always returning True"
  2. Push — git push origin fix/login-bug sends the branch to GitHub.
  3. Pipeline triggers — GitHub Actions detects the push and starts the workflow.
  4. Install — ./run.sh install installs dependencies on the runner.
  5. Test — ./run.sh test runs pytest. The new test for the login fix passes. All 12 tests pass.
  6. Build — docker build -t my-app:latest . builds a fresh image with the fix included.
  7. Smoke test — docker run --rm my-app:latest confirms the container starts.
  8. PR opened — the developer opens a pull request. The pipeline result appears as a green tick.
  9. Review and merge — a teammate reviews the code, approves it, and merges the PR into main.

This is DevOps in action. The whole process — from fix to confident merge — took minutes, not days.

DevOps culture in practice

The tools you have learned are only half the story. DevOps is equally about how teams work together.

Key cultural principles that professional teams live by:

  • Shared ownership — nobody says 'that is the ops team's problem'. If it runs in production, everyone is responsible.
  • Blameless post-mortems — when something breaks in production, the question is 'what failed in the system?', not 'who made the mistake?'. Fear of blame slows teams down.
  • Small, frequent changes — large releases are risky. Smaller changes are easier to test, easier to review and easier to roll back.
  • Automate everything repetitive — if you do it twice, script it. If you script it, version control it.
  • Measure and improve — track how long deployments take, how often pipelines fail, how quickly you recover from incidents.

What comes next

This module covered the foundational layer of DevOps. The field extends significantly further:

Container orchestration
Managing many containers across many servers. Kubernetes is the dominant tool.
Infrastructure as Code
Defining cloud infrastructure in code with tools like Terraform or Pulumi.
Observability
Understanding what your running application is doing via logs, metrics and traces.
DevSecOps
Embedding security checks into the pipeline — scanning for vulnerabilities and leaked secrets.
GitOps
Using the Git repository as the source of truth for infrastructure state.
SRE
Site Reliability Engineering — Google's operationalisation of DevOps principles.

These are appropriate topics for Year 2 and 3 modules, internships, and industry certifications. The foundation you have built this term is the prerequisite for all of them.

Exam preparation guide

The exam is two hours and covers all seven MLOs. Common question formats:

Command questions — 'write the Git command that creates a branch called feature/search and switches to it.' Answer: git switch -c feature/search

Trace questions — 'a developer runs the following commands. What is in the staging area after step 3?' These test your understanding of Git's three areas.

Concept questions — 'explain what a quality gate is and why it is used.' Two or three sentences is sufficient. Include an example.

Scenario questions — 'a pipeline is failing. The log shows pytest: command not found. What is the most likely cause and how would you fix it?' These test practical understanding.

Read the question carefully

Many exam mistakes come from answering the question you expected rather than the one that was asked. Take 30 seconds to identify exactly what is being asked before writing.

Key terms reference

A consolidated glossary of the most important terms from all ten modules:

DevOps
Practices unifying software development and IT operations.
CALMS
Culture, Automation, Lean, Measurement, Sharing.
shell
A program that reads and executes commands you type.
pipe (|)
Connects two commands — output of one becomes input of the next.
grep
Search a file for lines matching a pattern.
repository
A project tracked by Git, containing a .git directory.
commit
A snapshot of your project at a point in time.
branch
A separate line of development within a repository.
pull request
A proposal to merge one branch into another.
CI
Continuous Integration — automatically checking every code change.
pipeline
The automated sequence of steps triggered by a push.
quality gate
A check that must pass before code can merge.
Docker image
A read-only blueprint for a container.
container
A running, isolated instance of a Docker image.
Dockerfile
Instructions for building a Docker image.
smoke test
A minimal check that the application starts without crashing.
