Collaborating with Git

Learn to work with branches, remote repositories and pull requests — the everyday workflow of professional development teams.

MLO3MLO4MLO7

Why branches?

Imagine you are working on a project and you want to try a new feature — but you are not sure if it will work. You do not want to break the code that is already working.

A branch is Git's solution. It is a separate line of development. You can experiment freely on a branch without affecting the main code. If the experiment works, you merge it in. If it does not, you discard it.

The main branch

By convention, the primary branch of a repository is called main (historically master). Your stable, working code lives here. You should almost never commit directly to main — instead, create a branch, work there, and merge back.

Creating and using branches

bash — branch basics
# See all branches (the * shows your current one)
git branch
* main
# Create a new branch
git branch 
add-login-page
# Switch to it
git switch 
add-login-page
Switched to branch 'add-login-page'
# Create and switch in one step
git switch 
-c 
add-login-page

Now any commits you make are on add-login-page, not on main. When you are ready to bring the work back:

bash — merging
# Switch back to main first
git switch 
main
# Merge the feature branch into main
git merge 
add-login-page
Updating a1b2c3d..f4e5d6c
Fast-forward

Remote repositories

So far all your Git history lives only on your computer. A remote repository is a copy hosted somewhere else — usually GitHub. Remotes enable:

  • Collaboration — other people can access and contribute to the same repository
  • Backup — your code is safe even if your laptop fails
  • CI/CD — pipelines trigger automatically when you push to a remote
bash — working with remotes
# Clone an existing repo from GitHub
git clone 
https://github.com/user/repo.git
# See configured remotes
git remote 
-v
origin  https://github.com/user/repo.git (fetch)
origin  https://github.com/user/repo.git (push)
# Push your branch to GitHub
git push 
origin add-login-page
# Pull the latest changes from the remote
git pull

The pull request workflow

A pull request (PR) is a proposal to merge one branch into another. It is the standard way teams review and discuss code before it reaches the main branch.

  1. Push your branch to GitHub: git push origin your-branch
  2. Open a PR on GitHub by navigating to the repository and clicking 'Compare & pull request'.
  3. Write a description of what your changes do and why.
  4. Team members review the code, leave comments, and request changes if needed.
  5. Once approved, merge the PR — GitHub merges the branch into main.

Merge conflicts

A merge conflict happens when two branches have changed the same line of the same file. Git cannot decide which version to keep, so it asks you.

bash — a conflict looks like this
<<<<<<< HEAD
print('Hello, world')
=======
print('Greetings, world')
>>>>>>> add-greeting-branch

The section between <<<<<<< HEAD and ======= is what your branch has. Between ======= and >>>>>>> is what the other branch has. You must edit the file to keep what you want (delete all the marker lines), then stage and commit.

Conflicts are normal

Every developer encounters merge conflicts. They look alarming the first time but are straightforward to resolve once you understand the format. The key is to read both sides carefully before deciding which to keep.

Key terms

branch
A separate line of development within a repository.
merge
Combining the changes from one branch into another.
remote
A copy of the repository hosted on another server, typically GitHub.
origin
The conventional name for the default remote repository.
pull request
A proposal to merge a branch, reviewed by teammates before merging.
merge conflict
When two branches have changed the same part of a file and Git cannot auto-merge.

Exercises

Lab exercises
  1. Clone the provided starter repository from GitHub.
  2. Create a branch named feature/your-name and switch to it.
  3. Make two commits on your branch — add a new file and modify an existing one.
  4. Push your branch to GitHub and open a pull request with a description.
  5. Simulate a conflict: your tutor will push a change to the same file. Pull it and resolve the conflict.
  6. Commit the resolved file and merge the PR.
  7. Extension: after merging, delete the feature branch both locally and on GitHub.
