Introduction to DevOps and the Unix Environment

Understand what DevOps is, why it exists, and take your first steps in the terminal.

MLO1MLO2

What is DevOps?

Imagine two teams building a product together. The development team wants to ship new features as fast as possible. The operations team wants the system to stay stable and reliable. For decades these two teams had conflicting goals — developers said move fast, operations said don't break things.

DevOps is the set of practices, tools and culture that brings these teams together. The name is simply a blend of Development and Operations. Instead of throwing code over a wall for someone else to deal with, DevOps teams take joint ownership of the whole lifecycle — building, testing, deploying and supporting software.

Why does this matter for you?

Even as a first-year student, you are about to start writing code that needs to run somewhere, be shared with others, and be maintained over time. DevOps skills — version control, automation, pipelines — are the foundations of every professional software role you might go into.

The CALMS framework

CALMS is the most widely cited model for understanding what DevOps means in practice. It stands for:

Culture
Teams share responsibility. Blame is replaced by learning. Failure is a signal, not a crime.
Automation
Repetitive manual tasks are automated. If you do it twice, script it.
Lean
Eliminate waste. Small, frequent changes are safer than large, rare releases.
Measurement
You cannot improve what you do not measure. Track deployment frequency, failure rate, recovery time.
Sharing
Knowledge and tools are shared openly across teams.
Reading the framework

You do not need to memorise CALMS as a checklist. Use it as a lens: when you see a DevOps practice, ask yourself which of these five principles it is serving.

Why the command line?

Most of the tools you will use in this module — Git, Docker, CI pipelines — are operated primarily from the command line. Graphical interfaces exist for some of them, but the command line is:

  • Universal — every server, every cloud environment, every CI runner uses it
  • Scriptable — you can automate command-line steps; you cannot easily automate clicking buttons
  • Precise — commands are exact and repeatable; GUIs are vague and vary between versions

The command line has looked almost the same since the 1970s. Learning it is a long-term investment that pays off across every operating system and tool you will ever use.

Common misconception

The command line is not just for sysadmins or 'hackers'. Modern software developers spend a significant part of their working day in a terminal. This is normal.

Your first terminal session

Open a terminal (on Linux or macOS) or Git Bash / WSL on Windows. You will see a prompt — a short line ending in $ or %. This is the shell waiting for your instruction.

Try these commands to get oriented:

bash — navigation basics
pwd
/home/yourname         # print working directory
ls
Documents  Downloads  Desktop
ls -l
drwxr-xr-x  2 you  staff  64 Jan 10 09:00 Documents
cd Documents
pwd
/home/yourname/Documents
cd ..
             # go up one level
cd ~
             # go to your home directory

The structure is always the same: the $ is the prompt (you do not type it), then the command, then any flags (-l), then any arguments.

To get help for any command, type the command name followed by --help, or use the manual: man ls.

Key terms

DevOps
Practices and culture that unify software development and IT operations.
CALMS
Culture, Automation, Lean, Measurement, Sharing — a framework for understanding DevOps.
Shell
A program that reads commands you type and executes them.
Terminal
The window you use to interact with the shell.
Prompt
The characters the shell displays to show it is ready for input.
Flag
An option added to a command, usually starting with - or --. Example: ls -l.

Exercises

Lab exercises
  1. Open a terminal and run pwd. What directory are you in?
  2. Use ls to list the contents of your home directory.
  3. Create a new directory called devops-labs using mkdir devops-labs.
  4. Navigate into it with cd devops-labs, then create a file: touch notes.txt.
  5. List the contents again — can you see notes.txt?
  6. Find out what ls -la shows that ls -l does not.
  7. Extension: use man ls to read the manual. What does the -h flag do?
