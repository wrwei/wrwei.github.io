Version Control and Git Basics

Understand why version control exists and master the core Git commands you will use every day.

MLO4

The problem without version control

Here is a scenario you have probably encountered. You are working on an essay or a project. At some point your folder looks like this:

project_v1.py
project_v2.py
project_FINAL.py
project_FINAL_v2.py
project_FINAL_ACTUALLY_FINAL.py

This is a version control system — a bad one. It has no record of what changed between versions, why it changed, or who changed it. If you share files with a partner, you will overwrite each other's work. If something breaks, you cannot easily go back.

Git solves all of this.

The snapshots model

Git's mental model is simple: it takes snapshots of your project at points you choose. Each snapshot is called a commit. Git remembers every snapshot you have ever taken, and you can go back to any of them.

You decide when to take a snapshot. A good rule of thumb: commit whenever you reach a point where the code works and does something meaningful.

One important idea

Git does not store the differences between files (though it can show you them). It stores the complete state of your project at each commit. This makes it fast and reliable.

Setting up Git

Before your first commit, tell Git who you are. This information is attached to every commit you make:

bash — initial configuration
git config 
--global 
user.name 

git config 
--global 
user.email 

# Verify your settings
git config 
--list

Core Git commands

These six commands cover the vast majority of day-to-day Git use:

  1. git init — start a repository

    bash
    mkdir 
my-project
 && cd 
my-project
git init
Initialized empty Git repository in .git/

    This creates a hidden .git folder that stores all of Gitâ€™s data. Your project is now a repository.

  2. git status — see what has changed

    bash
    git status
On branch main
Untracked files:
  app.py
nothing added to commit but untracked files present

    Run this constantly. It tells you what Git knows about and what it does not.

  3. git add — stage changes

    bash
    git add 
app.py
# stage a specific file
git add 
.
  # stage everything

    Staging means telling Git: I want to include this in my next snapshot. You must stage before you commit.

  4. git commit — take a snapshot

    bash
    git commit 
-m 
"Add initial Flask app"
[main (root-commit) a1b2c3d] Add initial Flask app
 1 file changed, 12 insertions(+

    The -m flag lets you write the message inline. A good commit message is a short imperative sentence: Add login form, Fix null pointer in parser.

  5. git log — review history

    bash
    git log 
--oneline
a1b2c3d Add initial Flask app
9f3e2a1 Add requirements.txt
7d1c4b0 Initial commit

    Each line is one commit. The string on the left is its unique identifier.

  6. git restore — undo changes

    bash
    # Discard changes to a file (before staging)
git restore 
app.py
# Unstage a file (after git add, before commit)
git restore 
--staged 
app.py

    Use this when you made a mistake and want to go back to the last committed state.

A typical workflow

Every session with Git follows the same rhythm:

bash — daily workflow
1. Check what state you are in
git status
2. Make changes to your files
3. Stage the changes
git add 
.
4. Commit with a clear message
git commit 
-m 

5. Review your history
git log 
--oneline

Key terms

repository (repo)
A project folder tracked by Git, containing a hidden .git directory.
commit
A snapshot of your project at a point in time.
staging area
A holding area where you prepare changes before committing them.
working directory
The actual files on your disk that you edit.
HEAD
A pointer to the most recent commit on your current branch.
git status
The most important command — shows what has changed since the last commit.

Exercises

Lab exercises
  1. Create a new directory and initialise a Git repository inside it.
  2. Create a file app.py with a few lines of Python. Run git status — what do you see?
  3. Stage the file with git add app.py and run git status again. What changed?
  4. Commit with a meaningful message using git commit -m.
  5. Make two more changes to app.py and commit each one separately.
  6. Use git log --oneline to see all three commits.
  7. Use git diff to see what has changed since your last commit.
  8. Extension: make a deliberate mistake in the file. Use git restore to recover the previous version.
