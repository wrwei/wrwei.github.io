Continuous Integration and Automated Testing

Understand what CI is, what automated tests look like, and how to read pipeline output.

MLO1MLO2MLO3MLO5MLO7

The integration problem

Imagine ten developers all working on different features. Each person's code works perfectly on their own machine. On Friday they all try to merge into main at once. Half the code breaks.

This was the state of software development for decades. Integrating everyone's work was so painful it happened only a few times a year — and those were always stressful, bug-filled events. The problem even had a name: integration hell.

Continuous Integration (CI) is the practice of merging small, frequent changes — and automatically checking them every time.

The CI loop

Every time a developer pushes code to the shared repository, CI automatically triggers a pipeline:

  1. Push — a developer pushes a commit or opens a pull request.
  2. Build — the CI server checks out the code and runs any compilation or bundling steps.
  3. Test — the automated test suite runs against the new code.
  4. Report — the pipeline reports pass or fail. If it fails, the developer is notified immediately and the change is blocked from merging.
The key insight

Fast feedback is the point. If you break something, you want to know within minutes — while the code is still fresh in your head — not days later when someone else discovers the problem.

A quality gate is a rule the pipeline enforces: code that fails any check cannot merge into the main branch. Common gates include: all tests pass, code style is clean, no obvious security issues.

What is a test?

An automated test is a small program that checks whether another program does what it should. A test has three parts:

Arrange
Set up the inputs and the state you need.
Act
Call the function or run the code you are testing.
Assert
Check that the output or result matches what you expected.

Here is a simple example in Python:

python — a simple test
# The function we want to test
def 
add
(a, b):
    return 
a + b

# The test
def 
test_add
():
    result = 
add(2, 3)
    assert 
result == 5
    # If result is not 5, this line raises an error

If the assertion passes, the test passes. If the assertion fails, the test fails and the pipeline stops.

Running tests

We use pytest, a popular Python testing tool. Install it with pip install pytest, then run:

bash — running tests
# Run all tests in the current directory
pytest
========================= test session starts ==========================
collected 3 items

test_app.py ...                                               [100%]

========================== 3 passed in 0.12s ==========================

A failing test looks like this:

bash — a failing test
pytest
FAILED test_app.py::test_add - AssertionError: assert 6 == 5
1 failed, 2 passed in 0.14s
Tests are only as good as what they check

A passing test suite does not guarantee the code is correct — it only confirms it behaves according to the tests you have written. Writing good tests is a skill developed over time.

Reading CI output

When a CI pipeline runs, it produces a log. Learning to read it efficiently is a key skill — you will spend time doing this throughout your career.

The most important things to find in a failing pipeline log:

  • Which step failed (build, test, lint?)
  • The exact error message (usually in red or prefixed with ERROR or FAILED)
  • The file name and line number where the failure occurred

Ignore the noise — there is a lot of output about setup and dependency installation. Scroll to the first error, not the last.

Key terms

continuous integration
The practice of merging and automatically checking code changes frequently.
pipeline
The automated sequence of steps that runs when code is pushed.
quality gate
A rule that must pass before code can be merged.
test
A small program that checks whether your code behaves correctly.
pytest
A Python tool for running automated tests.
assert
A statement that raises an error if a condition is not true.
